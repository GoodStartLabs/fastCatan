# Spec 3.1 Reproduction — Deviations

## Deviation D1: Stage-3 teacher generation stall on shard-28+ resume (2026-07-19)

### Symptom
The 12-worker stage-3 teacher generator (`repro_3_1_stage3_resume`) appeared wedged after its
23:21 UTC resume: no new shards for ~2h, audit log frozen at 31,076 B, yet all 12 workers
pinned at ~100% CPU (cputime ~= etime). The prior 14-worker run had produced shards 0-27 then
stalled at the same boundary.

### Investigation
- Launch args verified correct (out-dir = canonical dataset dir, correct ckpt, sims=512,
  ab-depth=2, `--queue-canonical-distill`). NOT a malformed relaunch.
- Faithful single-process repro of shard-28's seed (real net + `MCTSvsFixed`, sims=512):
  games terminate normally at 175-417 plies, ~13-48 s each under heavy CPU contention. No hang
  reproduced across the games sampled.
- The "wedged" run was NOT dead: shard mtimes show it atomically published shards
  28-31, 33-36, 38-39 in a burst ~2 h after resume — ~4-6x slower than the original
  ~29 min/shard.
- The audit log only *looked* frozen: `resume.py` stdout was block-buffered through `tee`
  (no `python -u` / `PYTHONUNBUFFERED`), so progress lines flushed in late bursts.

### Root cause
No confirmed hard infinite loop. Two compounding effects:
1. A minority of seat-0 games run far longer than the ~7-15 s norm (finite but pathological),
   collapsing shard throughput ~4-6x.
2. Block-buffered logging masked the slow progress ("going dark").

The forced-timeout stack frame captured during the repro was an ordinary forward pass
(`torch.nn.Linear` in `net.forward`, via `models/alphazero/mcts_vs_fixed.py`
`_expand`/`_simulate`/`choose`) — research-surface code, NOT frozen engine code
(`src/catan/*`, `include/*`, bindings). Because a true non-terminating frame could not be
isolated, no engine change was made.

### Minimal repro (for upstream MCTS investigation)
- shard_id 28 -> seed = 20260608 * 1_000_003 + 28 * 7919 = **20260669003556**
- Sample many games (n >> 7) at sims=512, ab_depth=2, leaf_eval=ab_value,
  ab_value_scale=8.6e7, ckpt `repro_3_1_il_ab_d2_640k_vpm_ep10/il_final.pt`; a fraction run
  pathologically long. A hard non-terminating decision, if one exists, was not caught in the
  sampled games — the observed failure is severe finite slowdown, not a proven hang.

### Fix (research surface: `models/alphazero/stage3_gen.py`)
Opt-in per-game wall-clock watchdog. When env `STAGE3_GAME_TIMEOUT_S > 0`, each game is armed
with `SIGALRM`; a game exceeding the cap is abandoned — its partial seat-0 records are rolled
back and the search object is rebuilt from clean state — and generation continues to the next
game. Every abandonment is logged:
`[stage3-watchdog] shard <id> game <i> exceeded <T>s (non-terminating); abandoned`.
Default (env unset / 0) = byte-identical to the original code path.
**Activate with `STAGE3_GAME_TIMEOUT_S=120` and `PYTHONUNBUFFERED=1`.**

### Caveats / follow-ups
- The wall-clock cap is non-deterministic at the boundary (a borderline game may be abandoned
  on a slow machine but not a fast one). Each shard has a single producer, so the cross-machine
  dataset merge is unaffected in practice; if strict per-shard reproducibility is required,
  replace with a deterministic ply cap (e.g. 40000 -> 1500).
- Abandonment is recorded in the log but not yet in per-shard `.npz` metadata; add an
  `n_skipped` scalar to the shard if dataset-level accounting is required.
- Team-lead design preference was a per-decision guard; per-game was chosen because the failure
  mode could not be isolated, and a per-game wall-clock cap catches both a looping search and a
  long-but-finite game, whereas a per-decision cap would miss the latter.
